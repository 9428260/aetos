# 프로젝트 명

공동주택 커뮤니티 에너지 거래·최적화 지능형 에이전트 플랫폼  
Autonomous Agentic Energy Trading & Optimization System for Apartment Community

# 배경 및 문제 정의

## 1. 현재 상황

공동주택 단지에는 태양광, ESS, 부하 설비가 함께 존재하지만 실제 운영은 통합 최적화보다 개별 설비 제어 수준에 머무는 경우가 많다. 잉여 전력은 외부로 단순 판매되거나 활용되지 못하고, ESS는 실시간 가격, 부하, 발전량을 반영한 전략적 운영보다 제한적 충방전에 그치는 경우가 많다. 또한 운영 기록, 전략 선택 근거, 성과 지표가 분산되어 있어 관리자가 단지 에너지 운영을 일관되게 판단하기 어렵다.

## 2. Pain Point

현재 시스템이 해결하려는 핵심 문제는 다음과 같다.

- 발전량, 부하, ESS 상태를 종합한 실시간 운영 의사결정 부재
- 전략 생성, 평가, 선택, 실행이 분절되어 자동화 수준이 낮음
- 운영 이력과 KPI가 체계적으로 누적되지 않으면 성과 분석과 재현이 어려움
- 자연어 기반 운영 요청, 정책 검증, 정산 관점의 확장성이 부족함
- 운영자 입장에서 왜 이 전략이 선택되었는가에 대한 설명 가능성이 낮음

## 3. 기회 요인

AETOS는 멀티에이전트 기반 워크플로우, A2A 협상 구조, 벡터 메모리, MCP 도구 연계, 대시보드 및 이력 관리 기능을 이미 갖추고 있어 공동주택 에너지 운영 플랫폼으로 확장 가능한 기반을 확보하고 있다. 가격, 부하, 발전 예측과 전략 최적화, 정책 검증, 실행 로그 누적이 하나의 시스템 안에서 연결되어 있어 실제 단지 운영형 서비스로 구체화하기에 적합하다.

# 프로젝트 목표

## 1. 정성적 목표

- 목표 1: 공동주택 에너지 운영 의사결정의 자동화
- 목표 2: 전략 생성부터 실행까지 연결되는 에이전트 기반 운영 체계 구축
- 목표 3: 운영 이력, KPI, 전략 메모리 축적을 통한 지속적 고도화
- 목표 4: 운영자 관점의 설명 가능성과 모니터링 가시성 확보
- 목표 5: API, MCP, 대시보드를 통한 확장형 서비스 구조 확보

## 2. 정량적 목표 (KPI)

| 성과 지표 | 목표 수치 | 측정 방법 | 현재 시스템 기준 |
|---|---:|---|---:|
| 운영 이력 저장 성공률 | >= 99% | episode 저장 로그/DB 확인 | 구축 중 |
| 전략 실행 자동화율 | >= 90% | run/dispatch 로그 분석 | 구축 중 |
| ESS 활용률 | >= 80% | dispatch 및 KPI 로그 | 시뮬레이션 기반 |
| 전략 선택 응답시간 | <= 5초 | workflow step timing | 측정 가능 |
| 거래/운영 성공률 | >= 95% | step_events 및 audit log | 측정 가능 |
| 운영자 가시성 만족도 | >= 4.5/5 | 실증 사용자 평가 | 미측정 |

# 핵심 기능 정의

## 1. 에이전트 구성

| Agent 명 | 역할 |
|---|---|
| Strategy Agent | 운영 전략 후보 생성 |
| Optimizer Agent | 후보 전략 개선 및 보정 |
| Auction/Clearing Logic | 전략 간 경쟁 및 청산 판단 |
| Critic Agent | 정책·제약 위반 전략 제거 |
| Dispatch Agent | ESS/시장/감발 setpoint 실행 |
| Deep Agent | 자연어 기반 운영 요청 처리 |
| Forecast Runtime | 가격·부하·발전 예측 제공 |
| Memory Layer | 과거 전략/상태/보상 기억 및 재활용 |

## 2. 주요 기능

1. A2A 기반 전략 협상 및 선택  
전략 생성, 최적화, 경매, 비평, dispatch 단계가 워크플로우로 연결되어 자동 의사결정을 수행한다.

2. ESS 및 전력 운영 최적화  
ESS 충전·방전, 시장 거래량, PV 감발률, 부하 이동량을 함께 고려해 최적 전략을 선택한다.

3. 운영 이력 및 KPI 저장  
각 실행 사이클을 episode 단위로 저장하고 reward, cost saving, ESS profit, ROI를 누적 관리한다.

4. 전략 메모리 및 재활용  
벡터 메모리를 통해 과거 유사 상황의 전략과 성과를 검색해 후속 의사결정 품질을 높인다.

5. 자연어 운영 인터페이스  
`/chat` 기반 deep agent를 통해 운영자가 자연어로 최적화 요청이나 질의를 수행할 수 있다.

6. MCP 도구 연계  
forecast, optimize, policy_check, dispatch, kpi 도구를 외부 클라이언트와 연동할 수 있다.

7. 통합 모니터링 대시보드  
실행 로그, 전략 후보, 선택 결과, KPI, episode history를 UI에서 조회할 수 있다.

# 기대 효과

## 1. 경제적 효과

- 단지 에너지 운영의 손실 최소화 및 수익 기회 확대
- ESS 활용 최적화를 통한 투자 효율 개선

## 2. 운영적 효과

- 반복적 판단 업무 자동화
- 관리사무소 및 운영자의 의사결정 부담 감소
- 실행 이력과 KPI 기반 사후 분석 체계 확보

## 3. 기술적 효과

- 멀티에이전트 에너지 운영 아키텍처 실증
- A2A, MCP, 벡터 메모리 결합형 운영 플랫폼 확보
- 향후 공동주택 P2P 전력거래 서비스로 확장 가능한 기반 마련

## 4. 서비스 효과

- 대시보드, API, 자연어 인터페이스를 통한 운영 접근성 향상
- 운영 근거와 결과를 함께 제공하는 설명 가능한 시스템 구현

# 범위 및 제약사항

## 1. In Scope

- 단지 에너지 상태 기반 전략 생성·최적화
- ESS/시장/감발 중심의 dispatch 계산
- episode history 및 KPI 저장/조회
- 자연어 인터페이스 및 MCP 연계
- 운영 대시보드 및 로그 시각화

## 2. Out of Scope

- 실제 외부 전력시장 정산 연계
- 국가 전력망 직접 제어
- 현금성 개인 간 실거래 정산
- 대규모 상업용 발전소 수준의 실운영 제어

## 3. 제약 사항

| 구분 | 내용 |
|---|---|
| 제도 | 실제 P2P 전력거래 및 정산은 규제 검토 필요 |
| 기술 | 현재 일부 기능은 시뮬레이션/가상 상태 기반 |
| 인프라 | PostgreSQL/pgvector, API, loop 실행 환경 필요 |
| 보안 | API 키, 운영 로그, 사용자 요청 보호 필요 |
| 운영 | DB 연결, 이력 저장, vector 확장 등 배포 안정성 확보 필요 |
